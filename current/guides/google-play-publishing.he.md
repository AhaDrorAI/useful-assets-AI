# מדריך להעלאת אפליקציה ל-Google Play Store

מדריך מקיף למפתחים — עם דגש מיוחד על מפתחים עצמאיים ועוסקי פטור.

---

## מי המדריך הזה בשבילו

המדריך נכתב למפתח שמוכן לפרסם את האפליקציה הראשונה שלו, אבל עדיין לא עשה זאת. אם כבר פרסמת בעבר וצריך רק עזרה בשלב ספציפי — קפוץ ישירות לפרק הרלוונטי.

---

## שלב 1: פתיחת חשבון Play Console

כתובת: https://play.google.com/console/

### עלות

תשלום חד-פעמי של $25 בכרטיס אשראי. אין תשלום שנתי.

### בחירת סוג חשבון

בעת ההרשמה תתבקש לבחור בין שני סוגי חשבון:

| סוג | מתאים ל |
|---|---|
| אישי (Personal) | מפתח עצמאי שלא רוצה להירשם כארגון |
| ארגוני (Organization) | חברה, עמותה, או מפתח עצמאי עם מספר DUNS |

**המלצה למפתחים עצמאיים:** שקול ברצינות להירשם כ-Organization. ראה שלב 2.

## שלב 2: מספר DUNS למפתחים עצמאיים

זהו אחד הטיפים החשובים ביותר במדריך הזה.

### מה זה DUNS

DUNS הוא מספר זיהוי עסקי בינלאומי שמנוהל על ידי Dun & Bradstreet. גוגל משתמש בו כדי לאמת ארגונים. מפתחים עצמאיים — כולל עוסקי פטור — יכולים לקבל מספר DUNS.

### למה זה שווה את הזמן

כאשר נרשמים כ-Organization עם אימות DUNS:
- גוגל עשוי לאפשר פרסום ישיר לפרודקשן מבלי לעבור מסלול Closed Testing של 20 בודקים למשך 14 יום.
- החשבון נראה יותר אמין בעיני גוגל.
- מאפשר הוספת עמיתים לחשבון בעתיד.

### תהליך קבלת DUNS

1. כנס לאתר Dun & Bradstreet: https://www.dnb.com/
2. חפש "D-U-N-S Number" ובקש מספר חינם.
3. מלא את הפרטים הבאים:
   - שם העסק (שם+שם משפחה או שם העוסק הרשמי)
   - כתובת פיזית
   - מספר טלפון
   - סוג פעילות
4. זמן עיבוד: כ-שבוע.
5. תקבל מספר DUNS באימייל.

### אזהרה חשובה

**עקביות נתונים היא קריטית.** השם והכתובת שהזנת ב-D&B חייבים להיות זהים לחלוטין לנתונים שתכניס ב-Play Console. אפילו הבדל קטן כמו "סמטת האורן" מול "סמ' האורן" גורם לעיכוב באימות.

## שלב 3: הכנה טכנית

### פורמט AAB (חובה)

אפליקציות חדשות חייבות להיות מוגשות בפורמט **Android App Bundle (AAB)**, לא APK. APK בלבד אינו מקובל לאפליקציות חדשות.

**בניית AAB ב-Android Studio:**

```
Build → Generate Signed Bundle / APK → Android App Bundle
```

**בניית AAB דרך Gradle:**

```bash
./gradlew bundleRelease
```

הקובץ ייווצר בנתיב:

```
app/build/outputs/bundle/release/app-release.aab
```

### Play App Signing (חתימת אפליקציה)

לפני העלאה ראשונה — הפעל Play App Signing:

```
Play Console → Release → Setup → App signing
```

**כיצד זה עובד:**
- גוגל מנהל את מפתח הפרסום (Release key).
- אתה שומר על מפתח ההעלאה (Upload key).
- אם תאבד את מפתח ההעלאה — ניתן לבקש איפוס דרך Play Console Support.

**אל תעלה** את הקובץ החתום ישירות עם keystore מקומי כמפתח פרסום — עבור אפליקציות חדשות Play App Signing הוא תהליך העבודה המומלץ.

### רמת API (Target SDK)

גוגל מחייב שהאפליקציה תעמוד ברמת ה-Target SDK המינימלית שלו. הדרישה מתעדכנת בכל שנה.

בדוק את הדרישה הנוכחית:
https://developer.android.com/google/play/requirements/target-sdk

דוגמה לקובץ `build.gradle.kts`:

```kotlin
android {
    compileSdk = 35
    defaultConfig {
        targetSdk = 35
        minSdk = 24
    }
}
```

### בדיקות לפני העלאה

```bash
# בדיקות אינטגרציה
./gradlew connectedAndroidTest

# בדיקת Lint
./gradlew lint

# וידוא שה-AAB נבנה בהצלחה
./gradlew bundleRelease
```

## שלב 4: מדיניות פרטיות ו-Data Safety

אלו שני הרכיבים שגורמים לדחיות הכי הרבה. חשוב לטפל בהם בצורה מדויקת.

### מדיניות פרטיות

**דרישות:**
- חייבת להיות נגישה בכתובת URL פתוחה לציבור (לא קובץ מצורף).
- חייבת לפרט: מה נאסף, איך משתמשים בנתונים, עם מי משתפים.
- אפשר להשתמש בגנרטורים חינמיים כמו PrivacyPolicies.com.

**היכן להכניס ב-Play Console:**

```
Store listing → Privacy Policy → הכנס URL
```

### Data Safety Section

**מיקום ב-Play Console:**

```
Policy → App content → Data safety
```

זהו הטופס המפורט שמתאר לאיזה נתונים האפליקציה ניגשת.

**צ'ק-ליסט למילוי Data Safety:**

- [ ] כל סוגי הנתונים שנאספים (מיקום, מזהה מכשיר, אימייל וכו')
- [ ] האם הנתונים משותפים עם צד שלישי
- [ ] האם איסוף הנתונים הכרחי או אופציונלי
- [ ] האם נתונים מוצפנים בתעבורה ובאחסון
- [ ] פרטיות של כל SDK צד שלישי (Firebase, Ads, Analytics וכו')

**שגיאה נפוצה:** להצהיר "לא נאסף מידע" כאשר האפליקציה כוללת Firebase Analytics שכן אוסף מידע. גוגל בודק זאת ודוחה את הבקשה.

## שלב 5: עמוד החנות (Store Listing)

### שדות חובה

| שדה | מגבלה |
|---|---|
| שם האפליקציה | עד 30 תווים |
| תיאור קצר | עד 80 תווים |
| תיאור מלא | עד 4,000 תווים |
| אייקון | 512×512 פיקסל, PNG, ללא שקיפות |
| Feature graphic | 1024×500 פיקסל (חובה מ-2021) |
| צילומי מסך | לפחות 2 לטלפון |

### עצות לעמוד חנות

- **שם:** השתמש בשם שמשתמשים יחפשו — לא שם פנימי של פרויקט.
- **תיאור מלא:** כתוב בשפה הטבעית של הקהל שלך. אל תמלא רק מילות מפתח.
- **צילומי מסך:** חייבים לייצג את המסכים האמיתיים בגרסה הנוכחית. Mockup שמציג תכונה שעדיין לא קיימת — הוא סיבת דחייה.
- **Feature Graphic:** נדרש גם אם לא משתמשים בו לפרסום. בחר תמונה נקייה שמייצגת את האפליקציה.

## שלב 6: דירוג תוכן (Content Rating)

**מיקום ב-Play Console:**

```
Policy → App content → Content rating
```

מלא את שאלון IARC (International Age Rating Coalition) בכנות. דירוג לא מדויק גורם לדחייה ועלול להוביל להגבלת חשבון.

**שגיאה נפוצה:** לסמן "ללא אלימות" במשחק שמכיל קרב או ירי — גם אם הסגנון מצויר.

## שלב 7: מסלולי בדיקה (Testing Tracks)

גוגל דורש מעבר דרך מסלולי בדיקה לפני פרסום לפרודקשן.

### סדר המסלולים

```
Internal Testing → Closed Testing → Open Testing → Production
```

### Internal Testing

- עד 100 בודקים.
- ללא סקירה של גוגל.
- הפצה מיידית.
- מטרה: בדיקות עשן ראשוניות.

**הוספת בודקים:**

```
Release → Internal testing → Testers → Manage testers
```

### Closed Testing (Alpha)

- עבור חשבונות חדשים שאינם Organization: נדרשים לפחות **20 בודקים שהסכימו (opt-in) ל-14 ימים רצופים**.
- הבודקים צריכים להתקין ולהשתמש באפליקציה בפועל — לא רק להצטרף לרשימה.
- לאחר 14 יום, Play Console יציג אפשרות לבקש גישה לפרודקשן.

**אם נרשמת כ-Organization עם DUNS:** גוגל עשוי לדלג על הדרישה הזו. בדוק ב-Play Console האם קיימת אפשרות לפרסם ישירות לפרודקשן.

**מציאת בודקים:**
- [Testers Community](https://www.testerscommunity.com/)
- קבוצות פייסבוק / Discord של מפתחי אנדרואיד
- חברים ובני משפחה עם אנדרואיד

### Open Testing (Beta)

- אופציונלי.
- כל משתמש יכול להצטרף.
- מומלץ לאפליקציות גדולות לפני פרסום מלא.

## שלב 8: הגשה וסקירה

### לפני הגשה — וודא:

- [ ] כל שדות הרשימה מלאים
- [ ] Data Safety תואם ל-Privacy Policy
- [ ] שאלון Content Rating מולא
- [ ] AAB עלה בהצלחה ועובר Pre-launch report
- [ ] האפליקציה לא קורסת בהפעלה ראשונה
- [ ] אין תוכן placeholder ("לדוגמה", "Lorem ipsum")
- [ ] האפליקציה מספקת פונקציונליות אמיתית (לא רק WebView של אתר קיים)
- [ ] כל ה-Permissions המוצהרים אכן בשימוש

### הגשה:

```
Release → Production → Create new release → Upload AAB → Review → Start rollout
```

### זמן סקירה

הגשה ראשונה: בדרך כלל **2–7 ימי עסקים**.
עדכונים: בדרך כלל **1–3 ימי עסקים**.

## שלב 9: סיבות דחייה נפוצות ואיך לתקן

| סיבת דחייה | פתרון |
|---|---|
| Thin content | הוסף תוכן ופונקציונליות אמיתית; הסר placeholder |
| Deceptive behavior | הסר תיאורים מטעים או צילומי מסך לא מדויקים |
| Misleading metadata | ודא שצילומי המסך תואמים את הגרסה הנוכחית |
| Dangerous permissions | הסר הרשאות שאינן בשימוש מ-`AndroidManifest.xml` |
| Data Safety mismatch | עדכן את הטופס לכלול נתונים שנאספים ע"י SDKים של צד שלישי |
| קריסה בהפעלה | תקן קריסות לפני הגשה מחדש |
| Privacy Policy חסר | הוסף URL נגיש לעמוד החנות |
| Content Rating לא מדויק | חזור על שאלון IARC בכנות |
| WebView בלבד | הוסף ערך מוסף מעבר לתצוגת אתר קיים |

## שלב 10: תחזוקה לאחר הפרסום

### ביצועים

**Android Vitals** (Play Console → Android Vitals) מציג:
- Crash rate: גוגל מצפה לפחות מ-1.09%
- ANR rate: גוגל מצפה לפחות מ-0.47%

חריגה מסף אלו עלולה להוביל להגבלת נראות האפליקציה בחנות.

### עדכוני SDK שנתיים

גוגל מחייב עדכון Target SDK בכל שנה. כישלון בעמידה בדדליין — האפליקציה הופכת לא זמינה להורדות חדשות (גרסאות ישנות שהתקינו ממשיכות לעבוד).

בדוק את לוח הזמנים הנוכחי:
https://developer.android.com/google/play/requirements/target-sdk

### תגובה לביקורות

- הגב לביקורות שליליות בתוך 48 שעות כשניתן.
- אל תסתר ביקורות — גוגל מעקב אחרי כך.
- תגובה מנומסת ומועילה ממירה לעיתים קרובות 1 כוכב ל-4 כוכבים.

### Staged Rollout לעדכונים

עבור עדכונים גדולים, השתמש ב-Staged Rollout:

```
10% → 50% → 100%
```

זה מאפשר לתפוס תקלות לפני שמשפיעות על כלל המשתמשים.

### מעקב אחרי אימיילים מגוגל

גוגל שולח אזהרות לפני אכיפה (Policy warnings). אל תתעלם מהם — הם עדיין מאפשרים תיקון לפני השעיה.

## טיפים מהשטח

1. **מספר DUNS שווה את השבוע:** עבור מפתחים עצמאיים, ה-DUNS חוסך לרוב את מסלול ה-20 בודקים/14 יום. זה שבוע המתנה שחוסך שבועיים של גיוס בודקים.

2. **בדוק Data Safety לפני הכל:** הצהיר על כל SDK שמותקן באפליקציה — Firebase, Crashlytics, Ads, כל דבר. אחרת תחכה לדחייה.

3. **שמור גיבוי של ה-Upload Keystore:** אם איבדת את ה-Upload Key של Play App Signing, התהליך כרוך בפנייה לגוגל עם הוכחות בעלות. זה ניתן, אבל לוקח זמן.

4. **הגש מוקדם כדי לגלות בעיות:** הגשה ל-Internal Testing אפשרית מהיום הראשון. עשה זאת מוקדם כדי לגלות בעיות טכניות לפני שמגיעים לסקירה הראשית.

5. **שמות עקביים:** ודא שהשם שמוצג בחנות תואם ל-`android:label` ב-`AndroidManifest.xml`. חוסר עקביות מחשיד.
